# Test examples & expected outputs

Sanitized/fake sample transcripts (from `app.ipynb`) and the structured output the
pipeline produces for each — one example per domain the system routes to.

| File | Domain | What it exercises |
|---|---|---|
| `ex1.txt` | `debt_collection` | Settlement split, promise-to-pay dates, tape disclosure + mini-Miranda, supervisor-approval escalation. **Sharp date test:** "the 15th of next month" → `2026-08-15`, "Friday" → `2026-07-03`. |
| `ex2.txt` | `sales` | Pro-plan decision pending approvals, **owner-less** procurement task (the "don't guess an owner" test), procurement/legal blockers, "early next week" / "end of the month" dates. |
| `ex3.txt` | `customer_support` | Duplicate-charge refund promised without authority on a shaky system, consent + identity verification, escalation. "tomorrow" → next day. |
| `ex4.txt` | `general_meeting` | Postgres-vs-Elasticsearch decision, owner-less "flaky CI" task, "this week Wednesday" (= anchor) and "end of next week" → `2026-07-10`. |

For each `exN.txt` there are two output files:

- **`exN_out.json`** — the full structured `MeetingIntelligence` object the app returns
  (tag, summary, decisions, action items with owners/dates, blockers, compliance
  observations, human-review queue, and `evidence` rebuilt from the cited lines).
- **`exN_out.md`** — the same result, human-readable.

All examples use the meeting date **2026-07-01 (Wednesday)** as the date anchor.

## How these were produced / how to reproduce

They were generated by running the actual workflow against each transcript:

```bash
# from the repo root
python test_examples/_generate.py
```

- With `OPENAI_API_KEY` set, it uses the LLM (`gpt-4.1`) — the committed outputs here were
  produced this way, so they are genuine end-to-end results, fully line-grounded.
- With no key (or `FORCE_MOCK=1`), it uses the deterministic mock engine instead.

Because the LLM is involved, exact wording (and occasionally whether an owner is named or a
point lands as a decision vs. an action item) can vary slightly between runs. Treat these as
**reference** outputs: the invariants that must always hold — every item cited to a real
line, dates resolved against the anchor, unclear/authority items escalated to human review —
are what the automated checks in [`backend/test_pipeline.py`](../backend/test_pipeline.py)
verify (deterministically, in mock mode).

## Using them as a manual validation checklist

Load a sample in the UI (or POST the transcript to `/api/analyze`) and confirm against the
matching `exN_out.md`:

- [ ] Routed to the expected domain.
- [ ] Relative dates resolved to the concrete dates noted above.
- [ ] Owner-less tasks stay unassigned (ex2 procurement, ex4 flaky CI) — not guessed.
- [ ] Every decision/action/blocker/observation shows evidence rebuilt from its cited lines.
- [ ] Judgment/authority/consent items appear under **Needs human review**.
